# Vanguard Phase 3 Plan — Adversarial Review, HITL, Dynamic MCP Skills, Budget Guardrails

> Execution: subagent-driven, task-by-task, `/simplify` before each commit, English everywhere, no AI attribution.

**Goal:** Lift Vanguard from a linear pipeline to a self-critiquing, cost-bounded, human-escalating system: adversarial review, a judged repair loop with human-in-the-loop, runtime knowledge/profiling/test tools over MCP, and hard budget guardrails.

**Architecture:** Build on the existing primitives (`prepareContext`/`runAgent`/`disposeContext`, `runStages` with per-stage `model`, `llmJudge`, the zero-dep MCP server, `usage`/`costUsd` on results). The only new core concept is a **freezable context**: a run can stop between stages and leave the sandbox + worktree + session alive, returning a resumable handle. HITL and budget-killing both reuse that.

**Decisions (from brainstorming):** adversary = different model + red-team prompt; HITL = pause + preserve + print `docker exec -it` (exit reason `needs_human`); budget = freeze + resume on raised limit (exit reason `budget_exceeded`); MCP skills = knowledge fetch + edge-test gen + perf profiling.

- Add `adversarialReviewStages()` and extend the canonical flow: `planImplementReviewStages()` gains an optional adversary+repair tail, or a new `planImplementAdversaryStages()`:

- The adversary uses a **different model** than the implementer (`model: 'opus'`) and a red-team system prompt. It does not edit files; it emits `<findings>` JSON.
- **prompt-architect note (red-team system prompt):**
  ```
  <role>Adversarial security and performance reviewer. Assume the change is guilty until proven safe.</role>
  <policy>You never edit files. You only report. Prefer false positives over missed vulnerabilities.</policy>
  <guidelines>Hunt for: injection, secret/PII exposure, auth/authz gaps, unsafe deserialization, path traversal, ReDoS, N+1 and quadratic patterns, unbounded growth, race conditions. Verify each claim against the diff.</guidelines>
  <task_instructions>Review the diff. Emit ONLY <findings>{...}</findings> per the schema, sorted by severity. Then write <promise>COMPLETE</promise>.</task_instructions>
  ```

## Feature 2b — Human-in-the-loop (judged repair loop)

### Task 2 — `runJudgedRepair` with 3-strike HITL
**Files:** `src/pipeline/judged-repair.ts` (+ test), exports.

- `runJudgedRepair(ctx, opts: { generate: PipelineStage; repair: PipelineStage; judge: Judge; agent; variables?; maxRejects?: number })`:
  1. Run `generate` (or take the prior stage's result).
  2. Loop: `judge.judge({ testCase, output: lastDiff })` → if `passed`, return completed. If rejected, run `repair` (consuming the verdict reason via `{{JUDGE_REASON}}`), increment a reject counter.
  3. After `maxRejects` (default 3) consecutive rejects, STOP: do not dispose; return a `FrozenRun` with `reason: 'needs_human'` and `ctx.sandbox.shellCommand()`.
- The caller prints the shell command so the human can `docker exec -it` into the live sandbox, inspect, and decide.
- **senior-architect note:** this is the first runner that returns `FrozenRun`. It must be paired with `disposeContext(ctx, { keep: true })` on the needs_human path. The happy path disposes normally.
- **prompt-architect note:** the repairer prompt gets `{{JUDGE_REASON}}` (why the last attempt was rejected) so each retry is targeted, not blind.
- Tests: fake judge rejecting 3× → returns `needs_human` with a shell command, sandbox NOT destroyed; judge passing on attempt 2 → completed.

## Feature 3 — Dynamic MCP skills

### Task 3 — Extend the MCP server with knowledge / edge-tests / profiling
**Files:** `src/mcp/typecheck-server.mjs` (+ `.d.mts`), `src/mcp/config.ts`, tests.

- Add tools to the zero-dep server (env-configurable commands, same pattern as typecheck/run_tests):
  - `search_knowledge` — grep a docs/KB directory (path via `MCP_KNOWLEDGE_DIR`, default `docs/`) for a query argument; return matching files + snippets. Deterministic doc lookup before coding.
  - `generate_edge_tests` — run a configured command (`MCP_EDGETEST_CMD`) or, default, return the current `git diff` framed as "write edge-case tests for these changes" context.
  - `profile` — run `MCP_PROFILE_CMD` (e.g. a benchmark) and return hotspots/output.
- `injectMcpServer(sandbox, { knowledgeDir?, profileCmd?, edgeTestCmd? })` wires the env + tool names; `MCP_TOOL_NAMES` grows accordingly.
- **prompt-architect note (tools over instructions):** the default system prompt gains one line: "Before writing code, call `search_knowledge` for relevant docs; do not guess APIs." The agent must be told *which tool* for *which need*.
- Tests: `handleRequest` for each new tool with a fake runner; `injectMcpServer` returns the expanded tool list and config.

## Feature 4 — Budget guardrails

### Task 4 — Cumulative budget killing
**Files:** `src/core/types.ts` (`RunOptions.maxCostUsd`), `src/pipeline/pipeline.ts`.

- `runStages` accumulates `result.costUsd` across stages. Before each stage, if `spent + (already known) >= maxCostUsd` (default from `RunOptions.maxCostUsd ?? 5`), STOP before running it: return `FrozenRun` with `reason: 'budget_exceeded'`, `spentUsd`, and the outcomes so far; keep the context alive (`disposeContext({keep:true})`).
- Resume = call again with a higher `maxCostUsd` (the session/worktree persist), so no tokens are reburned for prior stages.
- Tests: stages whose fake costs cross $5 → stops before the over-budget stage, returns `budget_exceeded` with the right `spentUsd`; under budget → completes.

### Task 5 — Smart caching instrumentation
**Files:** `src/core/types.ts` (`RunResult.cacheEfficiency?`), `src/agents/claude-code.ts`, docs.

- Surface cache efficiency: `cacheReadInputTokens / (inputTokens + cacheReadInputTokens)` on `RunResult` (and aggregate per pipeline). 
- **prompt-architect note:** order prompts for cache hits — stable prefix first (`<role>/<policy>/<guidelines>` system prompt, then task), volatile last (`{{PREVIOUS_DIFF}}`); reuse sessions (already done) so the cached prefix is reused across turns. Document the rule in `defaultSystemPrompt` usage.
- Tests: cacheEfficiency computed from a fake `result` usage.

## Verification (end to end, live)
- Adversary: run `planImplementAdversaryStages()` on a deliberately weak issue (e.g. "add an endpoint that reads a file by name") and confirm the adversary reports a path-traversal finding the plain reviewer missed.
- HITL: force `llmJudge` to reject (strict criterion) → confirm `needs_human`, sandbox alive, the printed `docker exec -it` works.
- Budget: set `maxCostUsd: 0.05` → confirm `budget_exceeded` after the first stage, worktree preserved, resume with higher limit continues.
- MCP: confirm the agent calls `search_knowledge` before coding (transcript shows the tool call).
